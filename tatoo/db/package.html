Das Paket enthält Klassen für das Serialisieren von Objekten in eine Datenbank oder wahlweise auch in XML-Dateien.

Intention

Es gibt bereits verschiedene Bibliotheken wie zum Beispiel Hibernate, die einem die Arbeit des Serialisierens abnehmen. Dennoch wurde die Entscheidung getroffen ein eigenes Framework zu schreiben. Das hat mehrere Gründe.

  • Die meisten Serialisierungs-Frameworks unterstützen keine serialisierung in XML-Dateien. Diese Funktionalität wird hier aber gebraucht, weil geplant ist, Armeelisten in XML-Dateien zu schreiben um eine Speicherfunktionalität bereitzustellen.
  • Ein eigenes Framework ermöglicht höhere Flexibilität. so ist z.B. geplant ein Versions-Konvertierungssystem zu schreiben, so dass eine gespeicherte Armeeliste aus einer älteren Version von tatoo in eine neuere Version eingelesen werden kann. Das ist mit herkömmlichen Serialisierungsframeworks nicht machbar.
  • Weil ich es kann.
    Es handelt sich bei tatoo um ein Spass-Projekt, welches in meiner Freizeit entsteht. Ich will die Technik hinter der Serialisierung verstehen und nachbauen auch um meine Fähigkeiten zu verbessern.

Bei der Planung der Schnittstelle zur Datenbank waren mir folgende Punkte wichtig:

  • Die einzelnen Objekte sollten nicht von sich aus wissen wie sie sich in die Datenbank schreiben können. Ich wollte in den Klassen keine Methoden wie save_to_XML() oder ähnliches sehen.
  • Die Datenbank im Hintergrund muss wenn möglich sogar zur Laufzeit austauschbar sein. Ob da nun eine h2, sqlite oder MySQL oder eben XML hinter liegt sollte egal sein.
  • Das ganze Konstrukt sollte als in sich geschlossenes Modul vorliegen, welches ohne das Gesamtsystem umschreiben zu müssen verändert oder sogar neu geschrieben werden kann.
  • Ein import von älteren Daten in eine neue Version muss möglich sein. Nur so werden gespeicherte Armeelisten nicht bei einem Update ungültig. Ausserdem ist ein Austausch zwischen zwei Computern möglich auf denen zwei verschiedene Versionen installiert sind.

Überblick

Dreh und Angelpunt der Serialisierung ist die Klasse DBFactory. Es handelt sich hierbei um ein Singleton. Die Factory wird beim Laden von tatoo per Konfigurationsdatei auf eine spezifische Datenbank oder einer XML-Struktur initialisiert und baut eine Verbindung dazu auf.
Diese Verbindung kann mit der Methode getConnection() abgefragt werden. Diese Methode liefert ein DBConnection Objekt zurück. Die Klasse DBConnection ist eine Abstrakte Klasse welche je nach verwendetem "Datenbanktreiber" unterschiedliche konkrete Objekte enthält. Bisher implementiert ist nur {@link tatoo.db.sql.SQLConnection} welche eine Implementierung einer Verbindung mit einer SQL-Datenbank darstellt. Generell enthält sie Methoden um Abfragen auf die Datenbank bereit zu stellen. Die Abfragen sind soweit von der konkreten Implementierung Abstrahiert, dass sie für jeden "Treiber" gleich aussehen.
Konkret bedeutet das, dass drei verschiedene Abfragen erstellt werden können:

  • Datamanipulation zur Manipulation (INSERT, DELETE usw) von Daten.
  • Datadefinition zur Definition von Daten (ALTER TABLE) und
  • Query zur Abfrage von Daten (SELECT)
Konkret handelt es sich um die drei abstrakten Klassen DataManipulation, DataDefinition und Query. Diese Klassen sind wiederum abstrakt und werden lediglich vom konkreten "Treiber" implementiert.
